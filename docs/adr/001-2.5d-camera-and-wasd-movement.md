# ADR-001: 2.5D Camera with Rotation and WASD Movement

**Status:** Accepted (implementation deferred)  
**Date:** 2026-05-28  
**Affects:** Client rendering, camera system, input handling, character models, equipment visibility

---

## Context

The current client uses a 2D top-down perspective with click-to-move controls. While this works for the Tier 0 foundation, the long-term vision requires:

1. **Visible equipped armor** — Players should see their gear on their character. A top-down 2D view makes this difficult to communicate.
2. **Camera rotation** — Full 360-degree camera rotation around the player character so they can inspect their gear and see the world from any angle.
3. **Direct movement** — WASD movement instead of click-to-move for a more responsive, action-RPG feel.

## Decision

The game should transition from pure 2D top-down to a **2.5D perspective** with:

- **Full camera rotation** around the player character (orbit camera)
- **WASD directional movement** as the primary movement input
- **Character models that display equipped gear** visually

## Implementation Plan (for when this milestone is reached)

### Camera System
- Replace the fixed Camera2D with an orbit camera (likely requires moving to Godot's 3D renderer or a 2.5D approach using 3D camera with 2D-rendered sprites/models)
- Camera rotation via mouse drag or Q/E keys
- Maintain zoom controls

### Movement
- WASD for directional movement relative to camera facing
- Remove or demote click-to-move (could remain as a secondary option)
- Update `Player.gd` input handling
- Server `move_player` reducer already accepts target position — client sends movement intent at tick rate

### Visual Approach Options (to be decided during implementation)
1. **3D models with 2D world** — Low-poly 3D characters on a 2D tilemap
2. **Full 2.5D** — Isometric/oblique sprites with rotation (pre-rendered angles)
3. **Full 3D** — Transition to Godot's 3D renderer entirely

### Equipment Visibility
- Character model/sprite must have attachment points for each equip slot
- Equipment rendering updates when `equipment_changed` signal fires
- Art pipeline needs to produce equipment visuals (potentially via AI generation)

## Consequences

- The A* pathfinding system in `Pathfinder.gd` may need adaptation for WASD (collision-based instead of path-based)
- The programmatic `_draw()` placeholder visuals will be replaced
- Asset pipeline needs to shift from flat 2D sprites to either 3D models or multi-angle sprites
- Server logic (reducers, tables) remains unchanged — movement is still position-based

## When to Implement

This change should be implemented when the project reaches the visual/presentation upgrade milestone, after the core gameplay loops (M1) are proven out. The simulation layer (server) is already decoupled from presentation (client), so this is a client-only change.
